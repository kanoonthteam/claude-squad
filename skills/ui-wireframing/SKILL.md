# UI Wireframing

## Purpose

Guide the Designer agent in creating clear, consistent ASCII wireframes that translate acceptance criteria into visual layouts. This skill covers box-drawing techniques, layout primitives, component annotation, responsive patterns, common UI patterns, interaction flows, and accessibility annotations.

Wireframes produced with this skill give developers a concrete visual reference before they write any code, reducing ambiguity and rework.

### Common Responsive Transformations

```
1. Sidebar â†’ Hamburger Drawer:
   Desktop: persistent sidebar (240px)
   Tablet:  collapsible sidebar (overlay)
   Mobile:  hamburger icon â†’ full-screen drawer

2. Horizontal Nav â†’ Bottom Tab Bar:
   Desktop: horizontal nav in header
   Mobile:  bottom tab bar (5 items max)

3. Grid Columns Collapse:
   Desktop: 4 columns
   Tablet:  2 columns
   Mobile:  1 column

4. Table â†’ Card List:
   Desktop: full table with columns
   Mobile:  each row becomes a card

5. Multi-Panel â†’ Tab Navigation:
   Desktop: side-by-side panels
   Mobile:  tabbed interface switching between panels

6. Inline Actions â†’ Action Sheet:
   Desktop: action buttons visible on row hover
   Mobile:  long-press or swipe â†’ action sheet
```

```markdown

|------------------|-----------------------------------|-----------------------------|
```

## Anti-Patterns

Avoid these common wireframing mistakes:

### 1. Implementation Details in Wireframes

```
BAD:  "Use a React Portal for this modal"
BAD:  "This should be a flex container with gap: 16px"
BAD:  "Use useState for the toggle"

GOOD: "Modal overlays page content, centered"
GOOD: "16px gap between cards"
GOOD: "Toggle switches between List and Grid view"
```

### 2. Missing States

```
BAD:  Only showing the "happy path" with data loaded
GOOD: Wireframing empty state, loading state, error state, and populated state
```

### 3. Ignoring Responsive Behavior

```
BAD:  Single desktop wireframe with no responsive notes
GOOD: Desktop wireframe + responsive behavior table + mobile wireframe for complex layouts
```

### 4. Over-Detailed Wireframes

```
BAD:  Pixel-perfect ASCII art trying to replicate a mockup tool
GOOD: Clear layout showing structure, hierarchy, and key interactions
```

### 5. Wireframing Non-UI Tasks

```
BAD:  Creating a wireframe for "Add database index on users.email"
GOOD: Skipping this task â€” no UI component
```

### 6. Missing Navigation Context

```
BAD:  Isolated page wireframe with no indication of how users arrive or leave
GOOD: Wireframe + navigation flow showing entry/exit points
```

### 7. Ambiguous Interactive Elements

```
BAD:  Text that could be a link or label: "View Details"
GOOD: Clearly marked: [View Details] for button, or underlined "View Details" for link
```

### 8. No Component Naming

```
BAD:  Wireframe with anonymous boxes
GOOD: Every region labeled with a component name for developer reference
```
